# KeyChecker — дизайн

Дата: 2026-07-13
Статус: утверждён (дизайн одобрен пользователем)

## Цель

Self-hosted веб-сервис для команды, который проверяет API-ключи LLM-провайдеров
на валидность и, где API это позволяет, показывает баланс/лимиты. Проверка
разовая и stateless — ключи нигде не сохраняются.

## Область (MVP)

- Провайдеры: OpenAI, Anthropic (Claude), OpenRouter, Perplexity, FAL.ai.
- Ввод ключей пачкой: по одному ключу на строку, провайдер определяется
  автоматически по префиксу, с возможностью указать вручную для неопознанных.
- Базовая аутентификация: общий токен через переменную окружения (сервис для
  доверенной команды, не публичный).
- Один Docker-образ: Go-бэкенд отдаёт статику собранного React/Vite фронта и
  `/api`.

Вне области MVP: сохранение ключей, история, мониторинг по расписанию,
многопользовательские аккаунты.

## Стек

- Backend: Go (стандартная библиотека `net/http`, параллельные проверки через
  `golang.org/x/sync/errgroup` с ограничением параллелизма).
- Frontend: React + Vite (TypeScript).
- Упаковка: один Docker-образ, Go встраивает статику через `embed`.

## Архитектура

```
Browser (React/Vite SPA)
        │  POST /api/check  { keys: ["sk-...", "pplx-..."] }
        │  Authorization: Bearer <token>
        ▼
Go HTTP-сервер
  ├── auth middleware — сверяет Bearer-токен из env
  ├── detector        — по префиксу определяет провайдера
  ├── check handler   — фан-аут по ключам (errgroup, лимит параллелизма, таймаут)
  └── providers/      — интерфейс Provider + реализация на каждый сервис
        openai, anthropic, openrouter, perplexity, fal
        ▼
   Внешние API провайдеров (только исходящие HTTPS к whitelisted-доменам)
```

Сервер держит статику SPA и `/api`. Без БД и стейта.

## Ключевая абстракция — Provider

Каждый провайдер реализует общий интерфейс; добавление нового провайдера — это
один новый файл плюс регистрация.

```go
type Provider interface {
    ID() string                       // "openai"
    Matches(key string) bool          // префикс: sk-ant-, pplx-, sk-or-, ...
    Check(ctx context.Context, key string) (Result, error)
}

type Balance struct {
    Amount   float64
    Currency string
    Limit    *float64 // nil если лимит неизвестен
}

type Result struct {
    Provider string
    Valid    bool
    Balance  *Balance // nil если API не отдаёт баланс
    Detail   string   // инфо о модели / причина невалидности
}
```

`Balance` заполняется только там, где API его отдаёт:

| Провайдер   | Проверка валидности        | Баланс                          |
| ----------- | -------------------------- | ------------------------------- |
| OpenAI      | GET `/v1/models`           | Недоступен по API               |
| Anthropic   | GET `/v1/models`           | Недоступен по API               |
| OpenRouter  | GET `/api/v1/key`          | Да (остаток/лимит)              |
| Perplexity  | лёгкий запрос к API        | Проверить; заполнить если есть  |
| FAL.ai      | лёгкий запрос к API        | Проверить; заполнить если есть  |

Для провайдеров без баланса в `Detail` пишем человекочитаемую пометку
«баланс недоступен по API».

## Поток проверки

1. Хендлер `POST /api/check` принимает `{ keys: []string }` (плюс опциональный
   явно указанный провайдер на ключ).
2. Для каждого ключа прогоняется детектор; неопознанный ключ даёт результат
   со статусом «unknown — укажите провайдера вручную».
3. Проверки запускаются параллельно с ограничением (например, 8 одновременно)
   и таймаутом на каждый внешний запрос.
4. Ответ — массив результатов **в порядке ввода**; частичные ошибки не роняют
   весь запрос.

## Безопасность

- Ключи существуют только в теле запроса и в памяти на время проверки: не
  логируются, не пишутся на диск, не кэшируются.
- В любых логах и в полях `Detail`/ошибках ключ маскируется (`sk-...abcd`).
- Исходящие запросы только по HTTPS и только к whitelisted-доменам провайдеров
  (защита от SSRF).
- Аутентификация: общий Bearer-токен из переменной окружения; запросы без
  корректного токена отклоняются (401). Токен не логируется.
- Если токен в окружении не задан — сервис не стартует (fail closed), чтобы
  случайно не поднять открытый инстанс.

## Тестирование

- Юнит-тесты детектора (сопоставление префиксов).
- Юнит-тесты парсинга ответа каждого провайдера на замоканных HTTP-ответах
  (`net/http/httptest`) — валидный ключ, невалидный ключ, ответ с балансом,
  сетевая ошибка/таймаут. Без реальных ключей и без реальной сети.
- Тест фан-аут-хендлера: сохранение порядка, обработка таймаутов, частичные
  ошибки.
- Тест auth middleware: 401 без/с неверным токеном, проход с верным.
- Реальные сетевые вызовы к провайдерам в CI не выполняются.
