# ADR-0001: Принять vibe-coding-guidelines как процесс разработки

## Status

Accepted

Date: 2026-07-13

## Context

KeyChecker создаётся с нуля. Нужен воспроизводимый, проверяемый процесс
агентной разработки: единый набор правил для агентов, явные гейты
качества и защита от небезопасных действий с секретами. Владелец указал
референс `github.com/kravtandr/vibe-coding-guidelines` как гайдлайн.

## Decision

We will принять AI-assisted development workflow из
`https://github.com/kravtandr/vibe-coding-guidelines`, зафиксировав
источник на неизменяемом коммите `2ff8d73878e839a5eafe65369cf8f29fd326a204`
(ветка `main`; опубликованного тега `v0.1.0` в репозитории нет, поэтому
источник записан как ad hoc, а не релиз). Устанавливаем корневой
`AGENTS.md` с каноническими командами, `CLAUDE.md`-адаптер (`@AGENTS.md`),
CI на GitHub Actions и этот ADR.

Канонические команды проекта:

- Lint: `gofmt -l . && go vet ./...`
- Tests: `go test ./...`
- Build: `go build ./... && (cd web && npm run build)`
- Verify: `gofmt -l . && go vet ./... && go test ./... && go build ./...`

## Alternatives

- Не устанавливать формальный процесс — отклонено: владелец явно
  запросил применение гайдлайнов, а проверяемые гейты снижают риск
  регрессий.
- Ждать публикации тега `v0.1.0` — отклонено: тег отсутствует, а `main`
  резолвится в конкретный неизменяемый SHA, чего достаточно для
  provenance.

## Consequences

Появляются обязательные гейты (lint, test, build) локально и в CI и
проектное правило безопасности по ключам. Обновление гайдлайнов —
явная ручная миграция (автосинхронизации нет). Merge protection остаётся
`BLOCKED` до настройки владельцем в настройках репозитория. Откат
дешёвый: удалить установленные артефакты.
